# Slot Machine — FSM Driven Infinite Reel (Unity)

Реализация классического бесконечного слота с упором на архитектуру, производительность и чистоту кода.
Проект демонстрирует грамотное применение **FSM**, **MVC**, **Object Pooling** и оптимизацию под GC.

---

## Механика

### Базовая логика

* Классический **infinite scroll** слот
* 3 видимых элемента
* Вертикальное движение вниз
* Зацикленный список символов

### Управление состояниями

Вся логика построена на **Finite State Machine**:

```
Init → Ready → Spinning → Stopping
```

* Переходы полностью контролируются FSM
* Нет скрытых флагов и «размазанных» состояний
* Поведение строго детерминировано

---

## Игровой флоу

### Start

* Запускает вращение
* Реалистичный разгон через tween-анимацию
* Состояние → `Spinning`

### Stop

* Активируется **ровно через 3 секунды**
* Запускает фазу контролируемой остановки

### Остановка (не линейная!)

1. Плавное замедление
2. Поиск ближайшего символа
3. Мягкий snap в центр
4. Лёгкий эффект упругости через `Ease.OutBack`

Результат — визуально «живой» и физически правдоподобный финиш.

---

## Визуальные эффекты выигрыша

При совпадении:

* Центральный символ:

  * Зацикленная пульсирующая анимация
* Фон:

  * Активируется `Particle System`

Чёткий визуальный фидбек без перегрузки сцены.

# Архитектура

## MVC

Чёткое разделение ответственности:

| Компонент      | Ответственность     |
| -------------- | ------------------- |
| `SlotItemView` | Только отображение  |
| `SlotReel`     | Математика движения |
| Controller     | Управление логикой  |

View не знает о механике вращения.
Reel не знает о визуальных деталях.
Связность минимальная.

---

## Object Pooling

* Иконки создаются **один раз**
* Во время скролла не вызывается `Destroy`
* Элементы просто перемещаются наверх
* Меняется только спрайт

Никаких лишних аллокаций в рантайме.

---

## No Magic Strings

Все строковые ключи FSM и EventManager вынесены в:

```
GameConstants (static class)
```

Преимущества:

* Защита от опечаток
* Централизованное управление
* Лёгкий рефакторинг

---

## Data-Driven Design

Настройка через инспектор:

* `visibleItemsCount`
* `itemHeight`
* `spacing`

Механика не зависит от хардкода.
Изменение количества видимых элементов не требует переписывания логики.

# Оптимизация

Проект специально оптимизирован против микро-фризов и лишней нагрузки на GC.

## Работа с Garbage Collector

### foreach → for

В `Update()` и `ProcessMovement()`:

* исключён boxing
* предотвращены скрытые аллокации

---

### DOTween Best Practice

* Используется `DOTween.Sequence()`
* Секвенции кэшируются
* При новом спине вызывается `Kill()`
* Нет «висящих» твинов в памяти

---

### Избежание Closure

В `StartFinalSnap`:

* Аккумулятор `_snapMovedDelta` вынесен на уровень класса
* Нет создания анонимного класса
* Нет скрытых heap-аллокаций

---

### Без строковой интерполяции в hot-path

В часто вызываемых логах исключено:

```csharp
$"Text {var}"
```

Это снижает лишние аллокации строк.
